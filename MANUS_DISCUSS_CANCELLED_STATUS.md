## Manus 的异议

### 核心论点：MVP 阶段 `cancelled` 状态不会被触发

当前 MVP 的 UI 中，**没有任何"撤回申请"的入口**：

- 首页卡片：只有"申请联系"按钮，没有"撤回"
- 个人中心（Profile.tsx）：只显示"收到的申请"，没有"已发出的申请"列表，更没有撤回按钮
- Contact.tsx（发送申请页面）：发送后跳转，没有撤回入口

因此，`cancelled` 状态在 MVP 阶段**只能通过代码直接写入**，普通用户操作路径中不会产生这个状态。

### 为一个不会被触发的状态写 UI 逻辑，有以下风险：

1. **无效代码**：写了逻辑，但用户永远不会看到，增加维护负担
2. **误导性**：如果未来"撤回"功能的设计发生变化（比如撤回后允许重新申请），这段代码就需要修改，反而增加技术债
3. **用户困惑**：如果因为某种 bug 导致 `cancelled` 状态意外出现，用户看到"已撤回"会完全不知道发生了什么

### Manus 的建议

**MVP 阶段**：Home.tsx 只处理 4 种状态的显示逻辑：

| 状态 | 首页按钮显示 |
|------|------------|
| `sent` | 等待回复（禁用） |
| `viewed` | 对方已查看（禁用） |
| `accepted` | ✓ 已接受（禁用，绿色） |
| `rejected` | 已拒绝（禁用，灰色） |
| `cancelled` | **不处理，降级显示"申请联系"按钮**（因为不会出现） |

对于 `cancelled` 状态，如果意外出现，降级显示"申请联系"按钮（即当作未申请处理），比显示"已撤回"更安全——用户可以重新申请，不会卡死。

**Pro 阶段**：当"撤回申请"功能正式上线时，再一起设计 `cancelled` 状态的完整 UI 和业务逻辑（包括冷却期、重新申请限制等）。
