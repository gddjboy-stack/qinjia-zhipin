## 📋 详细的协作步骤

### **第1-2步：Manus完成功能并推送**

**Manus的工作：**
1. 在本地完成功能开发
2. 确保代码编译通过（`pnpm tsc --noEmit`）
3. 本地测试验证
4. 推送到GitHub Public仓库

**关键点：**
- 代码必须是可运行的（至少编译通过）
- 提交信息要清晰说明改动内容
- 建议使用feature分支便于追踪

---

### **第3步：Manus生成链接清单**

**Manus生成的Markdown格式清单：**

```markdown
## 代码审查链接 - [功能名称]

### 改动说明
[简要说明本次改动的内容]

### 关键文件链接
- **ProfileDetail.tsx**: https://raw.githubusercontent.com/[用户名]/qinjia-zhipin/main/client/src/pages/ProfileDetail.tsx
- **Publish.tsx**: https://raw.githubusercontent.com/[用户名]/qinjia-zhipin/main/client/src/pages/Publish.tsx
- **DataContext.tsx**: https://raw.githubusercontent.com/[用户名]/qinjia-zhipin/main/client/src/contexts/DataContext.tsx
- **Home.tsx**: https://raw.githubusercontent.com/[用户名]/qinjia-zhipin/main/client/src/pages/Home.tsx

### 完整仓库
https://github.com/[用户名]/qinjia-zhipin

### 需要Claude特别关注的地方
- [ ] 字段映射是否正确？
- [ ] 隐私设置逻辑是否正确？
- [ ] 数据流是否清晰？
```

**优点：**
- 用户只需复制粘贴
- Claude能直接访问链接
- 清晰标注重点审查项

---

### **第4-5步：用户转发给Claude**

**用户的工作：**
1. 复制Manus提供的链接清单
2. 发给Claude
3. 说明需要审查的内容

**示例消息：**
```
Claude，请帮我审查这个功能的代码。以下是链接清单：

[粘贴Manus提供的清单]

请重点关注上面标注的几个问题。
```

---

### **第6-7步：Claude审查并返回意见**

**Claude的工作：**
1. 访问raw链接读取代码
2. 分析代码逻辑
3. 提出改进建议

**Claude返回的审查意见格式：**

```markdown
## 代码审查意见

### 发现的问题

#### 🔴 严重问题
1. **问题描述**
   - 位置：文件名第X行
   - 现象：[描述问题]
   - 影响：[影响范围]
   - 建议修复：[具体建议]

#### 🟡 中等问题
1. **问题描述**
   - ...

#### 🟢 建议改进
1. **改进点**
   - ...

### 总体评价
[总体代码质量评价]
```

---

### **第8-9步：用户转发给Manus，Manus修复**

**用户的工作：**
1. 复制Claude的审查意见
2. 发给Manus

**Manus的工作：**
1. 根据Claude的建议修改代码
2. 本地测试
3. 推送到GitHub
4. 回到第3步

## 💡 建议

### **建议1：使用GitHub Issues记录**

为了保持长期的记录，建议在GitHub上创建Issues来追踪：

```markdown
## Issue模板

### 标题
[功能名称] - 代码审查

### 描述
- 功能：[简要描述]
- 状态：待审查 / 审查中 / 修复中 / 已完成
- 关键文件：[列出文件]

### 审查意见
[Claude的审查意见]

### 修复进度
- [ ] 修复问题1
- [ ] 修复问题2
- [ ] 本地测试
- [ ] 推送到GitHub
```

**优点：**
- 形成永久记录
- 便于回溯历史
- 清晰的进度追踪

---

### **建议2：定期同步而非一次性审查**

**建议的节奏：**
- 每完成一个功能模块后，就推送到GitHub并发给Claude审查
- 不要等到全部完成再审查（容易积累问题）
- 及时发现问题，及时修复

**例如：**
1. 完成"隐私设置"功能 → 推送 → Claude审查 → 修复
2. 完成"通知设置"功能 → 推送 → Claude审查 → 修复
3. 完成"关于我们"功能 → 推送 → Claude审查 → 修复

---

### **建议3：编码规范文档**

建议在GitHub仓库中创建 `CODING_STANDARDS.md`，明确：

```markdown
# 编码规范

## TypeScript
- 所有函数必须有类型注解
- 避免使用 `any` 类型
- 使用 `interface` 定义数据结构

## React
- 使用函数式组件和Hooks
- 避免在render中创建新对象/数组
- 使用 `useMemo` 和 `useCallback` 优化性能

## 数据结构
- 字段命名要清晰（如 `childLocation` vs `parentLocation`）
- 避免混淆相似的字段
- 在DataContext中统一管理所有数据结构

## 隐私设置逻辑
- 默认显示所有信息
- 只有用户主动关闭才隐藏
- 隐藏时显示"隐私"代替

## 电话号码处理
- 隐藏中间4位，保留前3位和后4位
- 使用实际输入的号码，不要硬编码
```

这样可以**预防**很多问题，而不是等到审查时才发现。

### **建议4：Pull Request模板**

建议在GitHub中创建PR模板，要求Manus每次提交都说明：

```markdown
## PR描述

- [ ] 修改了组件逻辑
- [ ] 修改了样式

## 测试情况
- [ ] 本地测试通过
- [ ] TypeScript编译通过
- [ ] 没有console错误

## 需要Claude重点审查的地方
- [列出需要特别关注的地方]
```

## 🔄 协作周期建议

### **短期（每周）**
- 完成1-2个功能模块
- 推送到GitHub
- Claude审查
- 修复问题

### **中期（每月）**
- 整理编码规范
- 更新文档
- 回顾已发现的问题模式
- 调整协作流程

### **长期（升级Pro前）**
- 总结所有学到的最佳实践
- 为升级Pro做准备
- 确保代码质量达到生产级别
